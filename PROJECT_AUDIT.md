# PROJECT AUDIT

Обновляется по мере работ по сайту `kbr-mebel.ru` / локальному проекту `stidio_sov_mebely`.

## Общие правила работы

- Сохраняем текущий дизайн сайта. Новые страницы и блоки собираем из существующего визуального языка: те же контейнеры, кнопки, отступы, типографика и общая структура.
- Для фронтенда не редактируем собранные файлы в `public/css` и `public/js` вручную. Исходники лежат в `src/scss` и `src/js`.
- После изменений во фронтенде пересобираем ассеты командой `npm run backend`.
- Для Blade-шаблонов публичной части ориентируемся на компоненты в `resources/views/ssm/components` и окружение в `resources/views/ssm/environment`.
- Админка использует bootstrap-подобную сетку (`container-fluid`, `row`, `col-*`), а публичный сайт использует собственную сетку через `.container` и SCSS-миксин `@include grid()`. Эти подходы не смешивать в одной публичной странице.

## Сетка и SCSS

- Базовая сетка публичного сайта задается в `src/scss/_settings.scss`:
  - `.container { @include grid(); }`
- Сам миксин находится в `src/scss/mixins/_breakpoint.scss`.
- Есть вспомогательный grid helper для наложения колонок:
  - `src/js/components/grid.js`
  - конфиг колонок: `resources/grid/grid.json`
- При создании новых блоков сначала использовать существующие контейнеры и компоненты, а не вводить новые layout-паттерны без необходимости.

## Важные проектные замечания

- На живом сайте footer уже закрывает контактную задачу. Внутри category-страниц лучше не дублировать телефон, WhatsApp, email и карту отдельным SEO-блоком.
- Формулировки должны звучать естественно и коммерчески. Избегать искусственных фраз вроде "Работаем в текущем стиле сайта".
- Если меняется категория или SEO-структура, нужно проверять:
  - `routes/web.php`
  - `app/Http/Controllers/MainController.php`
  - `app/Models/Group.php`
  - `resources/views/gallery.blade.php`
  - `resources/views/ssm/components/design.blade.php`
  - `resources/views/ssm/components/category-benefits.blade.php`

- В dropdown текущая категория должна подсвечиваться через active-состояние, чтобы пользователь видел, где он находится.

- В preview главной добавлены несколько hero-концептов для выбора направления, отдельный CTA-блок с живой формой заявки и новый preview-footer вместо старого общего футера.
- Для preview уменьшена “скругленность” блоков: крупные радиусы сокращены, чтобы интерфейс выглядел более собранным и менее шаблонным.
- В качестве базового направления для новой главной выбран hero-концепт №3: категории сразу в первом экране, с акцентом на SEO и быстрый переход в разделы.
- Контент preview главной хранится отдельно в `storage/app/home-preview-content.json` через `App\Support\HomePreviewContent`.
- Из админки `admin/home-preview/edit` уже редактируются блоки `hero`, `faq`, `cta` и `footer` preview-страницы.
- Сам preview подключает управляемые данные через `$previewContent`, поэтому дальше новые блоки лучше сразу делать в такой же схеме, а не хардкодом в Blade.
- Редактор preview перестроен в одну цельную форму: это проще поддерживать, чем несколько карточек со скрытыми полями между собой.
- В админку выведены и блоки ниже hero: `metrics`, `categories_section`, `approach`, `showcase`.
- Для блока `showcase` теперь отдельно управляется набор category slug через админку, а не берется жестко из первых элементов галереи.
- Боевая главная `/` переведена на новую структуру через общий partial `resources/views/ssm/components/new-home.blade.php`.
- Preview `admin/home-preview` и реальная главная теперь используют один и тот же контентный каркас; preview отличается только наличием hero-lab для внутренних тестов.
- Галерея и category pages начали переводиться под новый визуальный язык без добавления новых текстовых SEO-блоков.
- Из `gallery.blade.php` убран нижний `category-benefits`, а сама `design.blade.php` теперь отвечает за hero-шапку раздела, chip-навигацию, новые карточки и нижнюю навигацию по разделам.
- Отправка заявок теперь идет через Laravel-контроллер `Message\StoreController` не только в БД, но и в Telegram, если заданы `TELEGRAM_TOKEN`/`TELEGRAM_CHATID` или `TELEGRAM_BOT_TOKEN`/`TELEGRAM_CHAT_ID` в `.env`.
- Старый `public/tel.php` найден как исторический хвост; новая логика форм на него больше не опирается.
- На формах уже стоят 3 антиспам-слоя: honeypot (`website`), минимальное время заполнения (`form_started_at`) и rate limit по IP в `Message\StoreController` (5 заявок за 10 минут).
- Админская страница заявок `/message` переведена из простой таблицы в рабочий inbox-интерфейс: поиск, сортировка по колонкам, пагинация, выбор количества записей, статистика и удаление заявок.
- Для удаления заявок используется отдельный `Message\DeleteController`, а список собирается через `Message\IndexController` с query-параметрами `search`, `sort`, `direction`, `per_page`.
- В админке унифицированы списки `Группы`, `Статьи`, `Слайды`, `Настройки страниц`: везде теперь есть поиск, сортировка, пагинация, статистика, явные кнопки редактирования и подтверждаемое удаление.
- `MainSet` оставлен одиночной страницей, но визуально переведен в карточку настроек и теперь после сохранения делает нормальный redirect с flash-сообщением.

  - `Post\UpdateController` больше не возвращает на `slider.index`
  - `Group/Slider/Seting` update-контроллеры теперь редиректят обратно в свои разделы с сообщением об успехе
  - `Seting\UpdateRequest` снова валидирует поле `page`
